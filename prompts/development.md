# Self-development application

You are running the Selvedge engine on its own development. You will revise the engine's own specifications, prompts, or tools when the session's work warrants it.

## Read

Read what this session's work requires. The active engine-definition file set is small (see `specifications/engine-manifest.md`). Beyond the engine itself, read:

- The most recent session close in `provenance/` to understand where the workspace was left.
- `applications/075-selvedge-restart/selvedge-problem-statement.md` — the operator's framing of where the engine is going. This is the brief for sessions 076 onward.
- Any open issues in `open-issues/index.md` that bear on the session's work.
- `engine-feedback/` records flagged for triage.

You do **not** need to read the full provenance back-catalogue. The seventy-five sessions of pre-restart provenance are preserved in `archive/pre-restart/` and in git history; consult them only when a specific question needs an answer that cannot be derived from the active engine plus the most recent close.

## Operate via the substrate (engine-v20+)

**The substrate is the only writable surface for session state.** Markdown files in `provenance/` are *exports*, not authored prose. Every session phase is recorded by submitting a structured row through `bin/selvedge submit <kind>`; the markdown view is regenerated by `bin/selvedge export --session <N> --write`.

The flow:

1. **Open the session.** `bin/selvedge submit session-open` with `{session_no, slug, mode, workspace_id, engine_version_at_open}`.
2. **Record the assessment.** `bin/selvedge submit assessment` with `{session_no, state, agenda: [...]}`. The `state` is one short atom; `agenda` is a list of one-line items. Atoms must be 8–240 characters, no newlines, no fenced code, no pipe-table syntax.
3. **Convene perspectives if the work warrants.** For deliberative work: `bin/selvedge submit deliberation-open`, then `submit perspective` per perspective (the perspective body is then decomposed into one `perspective-position` row plus several `perspective-claim` rows under closed `section_kind`s — typically by a separate capture subagent), then `submit deliberation-seal` with synthesis and `submit synthesis-point` rows for convergences/divergences/minorities.
4. **Record decisions.** `bin/selvedge submit decision-record` with `{session_no, kind, title, outcome_type, target_kind, target_key, supports: [...], effects: [...], alternatives: [...]}`. Each support is `{basis, claim, cite?}`; each effect is `{effect_kind, target?, target_descriptor?}`; each alternative is `{label, option, rejections: [{basis, reason, cite?}]}`. Bases come from closed enums; claims are atoms.
5. **Run the coding review loop on any code change.** `bin/selvedge submit review-finding` per finding, then `submit finding-disposition` to close out medium-or-higher findings before close. T-20 refuses session-close while open medium-or-higher findings remain.
6. **Submit close.** `bin/selvedge submit close-record` with `{session_no, summary, items: [{facet, text}, ...]}` covering `what_was_done`, `state_at_close`, `open_issues`, `next_session_should`, `engine_version`, `validator_summary`. Then `bin/selvedge submit session-close` with `{session_no, engine_version_at_close}`.
7. **Export and commit.** `bin/selvedge export --session <N> --write` materialises `provenance/NNN-<slug>/00-assessment.md`, `01-deliberation.md`, `02-decisions.md`, `03-close.md`, `04-review.md` from substrate rows. `bash tools/validate.sh` confirms substrate integrity. Stage, commit, push.

You do **not** author markdown files for session phases. If you need to draft text, draft it in conversation or in a scratch file outside `provenance/` and then submit it via the CLI. The substrate refuses prose-in-rows by construction — atoms are length-bounded, newline-free, and typed; multi-paragraph reasoning is decomposed into typed link rows (supports, effects, claims, rejections) before submission.

## Multi-agent deliberation

When the work changes how the methodology works (a kernel revision, a deliberation-pattern change, a validation-mechanism change, a workspace-structure change, a database-schema decision), follow `specifications/methodology.md` §When to convene multiple agents:

- At least one perspective from a model family different from the orchestrator's (different organisation; e.g., not Anthropic). Use the `codex` CLI for cross-family perspective generation.
- Each perspective states its position before seeing others, to prevent anchoring. Run perspectives in parallel, sealed prompts.
- At least one perspective is adversarial.
- Synthesis preserves dissent; the synthesis is not itself a decision; it feeds Decide.

After deliberation, a capture subagent reads the perspective bodies and decomposes them into `perspective-position` + `perspective-claim` substrate rows. The orchestrator does not author the decomposition by hand.

## Review

Two reviewer mechanisms apply (see `specifications/methodology.md` §When to review):

- **Coding review loop.** If the session produces, modifies, or deletes executable logic (Python under `selvedge/`, SQL migrations, shell logic under `bin/` or `tools/`, or any other artefact whose execution behaviour the change alters), invoke a reviewer subagent — a distinct agent, not the implementer — to audit the change. Address every medium-or-higher finding (fix or explicitly adjudicate with a substantive reason recorded as a `finding-disposition` substrate row), then re-invoke the reviewer. Repeat until the reviewer reports no medium-or-higher findings remain, or the loop reaches the four-iteration deadlock threshold defined in the methodology spec (in which case the session halts rather than closes). The Agent tool with `subagent_type=Explore` and an adversarial prompt is the default invocation. Version strings, banner text, comments, and documentation changes are out of scope for the loop.
- **Engine-definition close review.** If the session changes any engine-definition file (per `engine-manifest.md`), or a structural validation check warns or fails, a separate reviewer audits the close. This is a single-pass audit, not a loop.

A session that triggers both records both: the loop's iterations and the close audit.

## Cautions for self-development

- **Don't import ideas from outside the process.** If an insight came from a conversation, an external paper, or your own pretraining, surface it as a hypothesis to be deliberated, not as a commitment.
- **Don't silently re-propose what was rejected before.** Cite the prior rejection (by `decision_v2` alias) and explain what changed.
- **Don't accumulate ceremony.** Each addition to the engine should pay for itself in capability, not in pretty paperwork.
- **Don't trust your own judgment alone on changes to how the methodology works.** Convene cross-family perspectives.
- **Be willing to subtract.** The successor needs a periodic mechanism that asks whether the accumulated weight has become the system's biggest problem and has the authority to remove.
- **Don't author markdown for session state.** The substrate is the canonical store; markdown is a generated view.
- **Don't track counters in prose.** Counts come from `bin/selvedge query` against the substrate. Prose claims about counts are forbidden in atoms (the substrate rejects pipe-tables and fenced code by construction).

## Close

Leave the workspace coherent at close. If a piece of work cannot complete, submit the produced rows, record the blocker as an open issue (TODO: open-issue submit kind), and end cleanly rather than leaving work in an indeterminate state. Run `bin/selvedge export` and `bash tools/validate.sh`. Commit at git per the workspace's commit discipline (`CLAUDE.md` if present).

The session's close states what the next session should address — concretely, in one or two `next_session_should` close-state items, with the rationale.
