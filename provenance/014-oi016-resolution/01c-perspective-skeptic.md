## Q1. Selection criteria for a reference case

I will propose criteria, but I want the reader to hold one thing in mind throughout: every criterion I list below is a way of narrowing the set of reference cases such that the reference-validation exercise is *less dishonest than it otherwise would be*. None of them make it honest. The mechanism is being asked to manufacture a validation signal from a comparison between two texts, neither of which is a domain-actor report that the artefact functioned in its intended use. That is a different epistemic object. The criteria are damage-control, not validation-equivalence. Q7 is where I press that hardest; here I do the constructive work.

Proposed criteria:

1. **Verifiable provenance of the reference solution.** The reference must have a traceable origin — a named author, a publication date, a venue or archive — such that someone reviewing the exercise later can confirm the reference was fixed *before* the methodology ran. A reference without provenance is indistinguishable from one written after the fact to match whatever the methodology produced. Concrete threshold: reference must have been published or archived at least 12 months before Session 015, with an accessible URL or citation, and a snapshot of that URL committed into the session's provenance folder as an anti-drift witness.

2. **Constraint-set is separable from solution-text.** It must be possible to state the problem — the constraints, the context, the stakeholder setting — without the statement smuggling the answer. Operational test: a third party (or a fresh agent) reading only the constraint statement must not be able to produce the reference solution verbatim. If the constraint statement is so detailed that the solution is implied, the case has leaked and should be rejected. This is the anti-leakage twin of the usual data-science test-set discipline (named here because I am importing the idea from pretraining: *data leakage* as a concept from machine-learning evaluation practice).

3. **Non-canonical in the Claude/GPT training distribution.** Self-evident but hard. A "proven solution" that appears in a Wikipedia article, a standard textbook, or a top Google result for the obvious search query is *presumptively contaminated*. See Q4 for the harder version of this criterion and why it cannot fully be met for most interesting reference cases. Concrete threshold: before selection, run a search query of the form "how to design X given constraints Y" on at least one public web index and one LLM; if the reference solution appears on the first page of results or is recited by the LLM from a naive prompt, the case is disqualified.

4. **Staged constraint emergence is plausible.** Per user steering, the case must admit a believable stagger — some constraints knowable up front, others only discoverable through engagement with the problem (a failed prototype, a stakeholder response, a regulatory wrinkle that arrives mid-project). The reference solution's historical design process must itself have encountered emergent constraints, so that the exercise is not a retrofit. Operational requirement: when selecting a case, the selector must document *which* constraints were known at problem-kickoff and *which* arrived later, with evidence (project diaries, retrospective papers, change logs). Cases where all constraints were specified up front are weaker tests and should be deprioritised.

5. **Domain-legibility to the available judges.** The reference case must be in a domain where the judges (whatever shape Q3 takes) can *read* both the reference and the methodology's output with enough competence to distinguish a structural match from a lexical coincidence. If judges need domain expertise the agents don't have, the comparison collapses to surface-string overlap. This is a harder constraint than it looks: it rules out most deeply technical domains for a Claude-subagent judging panel, which pushes selection toward domains where Claude is *strong*, which is exactly the contamination-heavy regime.

6. **Bounded artefact size and judgeable granularity.** The reference must be an artefact of bounded length — a protocol, a specification, a short design document — not a multi-thousand-page engineering manual. Practical threshold: reference artefact fits in under 10,000 tokens; the methodology's output can be compared section-by-section against named structural elements of the reference. If the reference is too large to hold in context with the methodology's output, comparison becomes a retrieval-over-retrieval game and loses whatever epistemic value it had.

7. **Representativeness of what the methodology claims.** The methodology claims it can produce "artefacts intended for use outside the workspace" — sequences, protocols, specifications. The reference case must be one of these types, not a proof, not a mathematical construction, not a piece of music. A category mismatch would let the mechanism pass or fail for reasons unrelated to the claim being validated. This is a *construct validity* criterion (named as pretrained input: construct validity from psychometrics, the question of whether a test measures what it claims to measure).

8. **The reference is not itself co-designed with an LLM.** Newly-common: a "proven solution" that was itself produced with GPT-4 assistance, published 2023–2026, will be a solution that already reflects LLM design patterns. Comparing an LLM-methodology output to an LLM-co-designed reference is a short-circuit. Operational threshold: reference predates widespread generative-AI availability in its domain (for most domains, before 2022).

Criteria (4), (5), and (8) are in tension with (3). A domain legible to Claude subagents, whose reference pre-dates 2022 and whose constraint structure is staged, is almost by construction a domain heavily represented in Claude's pretraining. This tension is not mine to resolve; it is a property of the design space the user has narrowed to, and the synthesiser should know I am flagging it rather than glossing it.

## Q2. The comparison procedure

The comparison procedure has to do three jobs at once and I am not confident any specification does all three well. The jobs are: (a) prevent reference-leakage into the Produce step; (b) produce a legible pass/partial/fail signal at the end; (c) handle the staged-constraint requirement from user steering without letting the staging become its own form of information leakage. My proposed procedure:

**Pre-run phase (before Session 015 opens).** A separate agent, not participating in Session 015's Produce step, performs *reference selection and constraint extraction*. This agent reads the reference, identifies the full constraint set, separates those constraints into *initial* (knowable at project start) and *emergent* (discoverable through engagement), and stores the reference itself in a provenance folder that Produce-step agents are instructed not to read. The extracted constraint set is the only thing that enters Session 015's Read step. The extractor agent's work product is committed to the repo with a hash that can be audited later.

**Isolation discipline for Produce agents.** Session 015's Produce step runs with agents who have been given the initial constraint set only. They are explicitly instructed not to search for or retrieve the reference, and their context window is auditable from the session transcript. This is imperfect (see Q4) but it closes the dumbest failure mode, which is an agent literally fetching and paraphrasing the reference.

**Staged constraint release.** Emergent constraints are released to the Produce step at pre-specified trigger points — e.g., "after the first Deliberate round produces a draft artefact, the orchestrator releases emergent constraint E1"; "if the draft is reviewed and passes workspace-validation, release E2." The trigger schedule is fixed before Session 015 runs; the schedule itself is part of the committed provenance. This prevents the orchestrator from adaptively staging constraints to make the methodology's output look good or bad.

**Comparison phase.** Once the methodology produces a candidate artefact and the staged constraint release is exhausted, a separate *comparison agent* (or panel, per Q3) receives three inputs: the constraint set, the methodology's artefact, and the reference. It performs a *structural* comparison — not a string-match — across dimensions fixed in advance. Proposed dimensions: (i) coverage of the constraint set (does the methodology's artefact address each stated constraint?); (ii) structural correspondence (do the methodology's sections, moves, or steps map onto the reference's?); (iii) substantive divergences (where the artefacts differ, are the divergences defensible given the constraints, or are they errors?); (iv) novel content (does the methodology's artefact include material the reference lacks, and is that material defensible or noise?).

**Match taxonomy.** I reject a binary pass/fail. Proposed three-state result, with one additional label:

- **Structural match** — the methodology's artefact addresses the same constraints with a comparable structural shape, and divergences are defensible. Note: this is not "match" in the strong sense; two documents can share structure and differ on every substantive claim.
- **Partial match** — coverage is incomplete or the structural correspondence holds for some sections and breaks for others. Specific gaps are enumerable.
- **Substantive divergence** — the methodology's artefact takes a different approach to the constraints; the comparison agent must judge whether the divergence is *defensible* (a reasonable alternative design) or *erroneous* (the methodology missed something the reference got right).
- **Contaminated** — the comparison agent detects or suspects that the Produce agents had reference access, via stylistic tells, quotation, or suspiciously exact structural mirroring. A contaminated result is neither pass nor fail; it is *no-signal*, and the exercise must be re-run with a different reference.

**Iteration.** If the first comparison returns partial match or substantive divergence, the methodology is permitted one iteration: return to Read or Deliberate with the comparison-agent's findings as input, revise, re-compare. After the second comparison the result stands. More than one iteration risks turning the exercise into a training loop where the methodology learns to produce the reference, and I would rather have a documented fail than an over-iterated pass.

**Failure cases.** If after one iteration the result is still substantive divergence, the session closes with that result explicitly recorded. This is not automatic methodology-fail (see Q6c); it is the data point for whatever aggregated methodology-level claim is being built.

## Q3. Shape of the validation step

I argue for a **hybrid of (ii) iteration-within-single-session and (iii) hand-off to subsequent session**, with a specific role-split, and I explicitly reject pure multi-agent judging (i) as the primary shape.

Justification. The user's follow-up steer emphasises that the methodology must be able to *absorb emergent constraints while producing the artefact*. A multi-agent jury reviewing a finished artefact is structurally bad at this. Juries review outputs; they do not participate in the production process. If the validation concern is *can the methodology handle emergent constraints in flight*, the validation shape has to be one that can observe and intervene in flight, or at minimum retrospect with fidelity over the in-flight decisions. That points to iteration-within-session for the in-flight part and hand-off-to-subsequent-session for the retrospect.

Proposed shape:

**Within Session 015 (Produce + iterate):** the session runs Read, Deliberate, Decide, Produce, Validate with the staged constraint release from Q2. Within-session iteration is triggered when Validate surfaces a gap: the session returns to Read or Deliberate, absorbs the new constraint, revises, re-validates. The within-session validator is a single adversarial sub-agent whose role is to read the draft artefact against the known-at-this-point constraint set and flag coverage failures or internal contradictions. One agent, not a panel, because a panel at this stage over-processes drafts.

**At Session 016 (hand-off, reference-comparison):** the subsequent session performs the reference-comparison per Q2. It runs the comparison agent as a single-agent single-role task *plus* an adversarial-reader sub-agent whose sole job is to attempt to find contamination signals (Q4). The comparison agent produces the match-taxonomy label; the adversarial reader produces the contamination assessment. The session's output is the two-part finding.

Why not multi-agent panel at Session 016? Two reasons. First, multi-agent juries across a single model family (three Claude subagents in different "roles") are not giving independent signals; they are giving three samples of the same pretraining. Naming the pretrained input here: this is the *correlated-errors* problem from ensemble learning — averaging correlated estimators does not reduce bias, only variance. Second, the session's load is already substantial (comparison plus contamination assessment plus documenting findings); adding jury deliberation increases cost for marginal epistemic gain.

Cross-model participation. At least one of the comparison or contamination-assessment steps should be run against a *non-Claude* model (GPT-5 or similar via `codex exec`). This is not a convergence check in the naive sense; it is specifically for the contamination assessment, where cross-model divergence on the methodology's artefact is the main non-trivial contamination signal available (see Q4). The methodology budget should allow one cross-model call per reference-validation exercise.

Dissent preservation. If the within-session adversarial reader or the Session 016 adversarial contamination-reader holds a dissenting view, it is preserved verbatim in the session record, per the workspace's standing rule on minority preservation. If the comparison agent calls "structural match" and the contamination reader calls "contaminated," the result is recorded as *no-signal*, not as pass-with-dissent. Contamination dominates structural finding.

How emergent-constraint steer interacts with this shape. The within-session iteration handles emergent constraints by construction — the staged release is part of the session's run. The Session 016 comparison evaluates whether the methodology *did in fact* handle the emergent constraints, by looking at the decision-trace in the Session 015 transcript. So the hand-off session evaluates not just the final artefact but the in-flight record. This is load-bearing: the validation is of the *methodology's process* as much as the *artefact's content*.

## Q4. Defence against pretraining-contamination / circularity

This is the one. Everything else in this perspective is warm-up.

**The specific failure mode, named precisely.** The Produce agents receive the constraint set. A constraint set sufficient to specify a design problem is also, for any well-documented reference, sufficient to *identify* the problem. Modern LLMs do not perform blind inference from a constraint list to a design; they perform a hybrid of retrieval and inference, with retrieval dominant when training data covers the problem. The agent effectively thinks: "this is the X problem; canonical solutions to X include A, B, C; given the stated constraint set, A fits best; I will produce A (with variations suited to the constraint phrasing)." The mechanism then compares this retrieved-then-surfaced A to the reference, which is also A or near-A, and declares a structural match. The validation signal is *zero* — we have measured the overlap of the training distribution with the reference distribution, not the methodology's design capacity. This is the retrieval-masquerading-as-design failure.

**The operational observable — how would anyone know.** This is where I press hardest. From a session transcript, how do we distinguish (a) "agents recognised the problem, retrieved the canonical solution, surfaced it with methodology-flavoured rhetoric" from (b) "agents were handed constraints, worked them through the methodology's steps, arrived at something resembling the canonical solution because the canonical solution is genuinely constraint-optimal"? The transcripts of (a) and (b) are plausibly indistinguishable. An agent that has retrieved A can reconstruct a plausible derivation-trace of A from the constraints; that is what chain-of-thought generation does well. Naming the pretrained input: this is the *unfaithful reasoning* phenomenon — chain-of-thought traces do not necessarily reflect the model's actual computation; a model can produce a plausible derivation post-hoc for a conclusion it reached by retrieval.

So the honest answer to "how would anyone know" is: mostly, we can't. The defences below reduce the probability of (a) going undetected, but none eliminate it.

**Evaluating the defences the brief lists:**

*(i) Select references explicitly outside common training distributions.* Survives *partially*. This is the best available move and I listed it as Q1 criterion (3). Its limits: "outside common training distributions" is an empirical claim about the training data of models we don't have transparent access to. Even domains that feel obscure (say, a specific 1973 industrial engineering report) may have been scraped via an archival digitisation project. More importantly, the *class* of problem is almost always in the training distribution even if the specific reference isn't — an obscure document describing a standard problem invites retrieval of the standard solution, which is then compared against the obscure document's (presumably also standard-shaped) solution. *Survives adversarial pressure only for genuinely novel problem formulations, which are scarce.*

*(ii) Blind the Produce agents to the reference.* Survives the *dumb* failure mode (agents literally having the reference in context) but does not survive the sophisticated one (agents having the reference's content in their weights). This defence is necessary but not sufficient. *Partial survival.*

*(iii) Use cross-model divergence as a contamination signal.* Survives *if and only if* the cross-model comparison is well-designed. The naive version — Claude and GPT-5 both produce the reference word-for-word — catches only egregious contamination; reference texts are paraphrased across training data, and both models are trained to paraphrase rather than quote. A more sophisticated version uses cross-model divergence at the *structural* level, which has more signal, but is still defeated when both models are trained on overlapping corpora (which they are). Additionally, *cross-model convergence on structural features is also the expected outcome when the reference is genuinely the constraint-optimal solution* — i.e., convergence is ambiguous between contamination and correctness. *Survives as a useful signal only, not as a decisive one.*

*(iv) Structural rather than lexical match.* Survives *worst of the four*. A structural comparison is more forgiving of surface differences and more likely to call "match" on a retrieved solution dressed up in methodology rhetoric. If lexical match suggests copying, structural match suggests the much harder-to-catch case of copying the *shape* of a solution while varying the words. This defence makes contamination *easier to miss*, not harder. I would argue against structural comparison being the primary criterion. *Does not survive adversarial pressure; I reject it as the load-bearing comparison level.*

*(v) Something else* (my addition): **Constraint-counterfactual probing.** After the Produce step completes, the orchestrator modifies one or more constraints to be *off-distribution* for the reference — e.g., flip a parameter such that the canonical solution no longer applies — and asks the methodology to re-produce. If the methodology produces a *genuinely different* artefact that tracks the new constraint, that is evidence of design responsiveness rather than retrieval. If the methodology produces a near-copy that ignores the constraint change, that is retrieval evidence. This is the strongest defence I can construct; it is also the most expensive. *Survives adversarial pressure partially; is the only defence that actively probes design vs retrieval rather than passively measuring output overlap.*

**The regime in which no defence is adequate.** For classical engineering problems with canonical solutions, widely-taught design patterns, textbook-standard protocols, and heavily-documented technical canon, reference-validation is not evidence of methodology capability regardless of mechanism. Specific examples of the regime: circuit design for a standard amplifier topology, recipe for a classical dish, protocol for a widely-taught medical procedure, design pattern for a common software architecture, project-management workflow for a category of project. In this regime, the methodology's output and the reference will both be drawn from the same dense cluster of canonical solutions; "match" is not evidence of design; "divergence" is not evidence against design (may be stylistic variation); "structural match with cross-model convergence" is the expected outcome whether or not the methodology is doing any work. The reference-validation exercise, in this regime, measures nothing about the methodology.

**Where this leaves the mechanism.** Any reference case selected per Q1 is *either* in the well-documented canonical regime (in which case the validation signal is near-zero for the reasons just listed) *or* outside it (in which case the reference itself becomes harder to trust as a "proven" solution — see secondary attacks on reference-solution-quality in Q7). The mechanism operates in a narrow sweet spot of references that are documented enough to be "proven" but obscure enough that retrieval doesn't dominate, and that sweet spot is much smaller than the brief suggests. I am pessimistic about whether it is non-empty in practice.

## Q5. Kernel §7 revision

My revision is more minimal than I expect the other perspectives to propose, because I think promoting reference-validation to first-class status in the kernel is precisely the "procedural self-deception" move the Session 013 Outsider warned against. My preferred revision keeps the two existing senses intact, adds a *provisional* third sense with explicit caveats, and requires the caveats be read aloud (so to speak) in every session that invokes it.

Proposed replacement for §7:

> **Workspace validation** occurs during or at the end of a session. Demonstrate that the produced or revised artefact, specification, or methodology segment satisfies the constraints it was produced under, is internally consistent, and fits the workspace's existing structure. Record the evidence used.
>
> **Domain validation** applies when a session produces or revises an artefact intended for use outside the workspace. Obtain evidence from the domain-actor who holds the problem the artefact addresses — the user of a sequence, the reader of a specification, the participant in a process, or an equivalent — that the artefact functioned for its intended use. Record who performed the validation, what was tried, what happened, and whether modifications were requested. Domain validation may complete out-of-session; when it does, the receiving session records the report and decides whether it triggers revision.
>
> **Reference validation** is a *provisional substitute* for domain validation, usable only when no domain-actor is available and the alternative is producing an artefact without any external check. Select a use case with a documented proven solution whose provenance predates the session by at least 12 months; extract the constraint set from the reference; run the methodology's Produce step on the constraint set, isolated from the reference; compare the methodology's artefact to the reference using the procedure specified in [mechanism spec]; record the comparison result, explicitly including any contamination signal detected. Reference validation does **not** establish that the artefact functioned in its intended use; it establishes only whether the methodology, on a known-solved problem, produced an artefact structurally comparable to the known solution. A session that relies on reference validation must state so, must not claim domain validation, and must preserve a dissent record if any participant holds that reference validation is inadequate for the specific artefact in question.

Three things to notice about this text. First, the language "provisional substitute" is deliberate: I refuse to let reference-validation be named alongside domain-validation as an equal third pillar, because it isn't one. Second, the paragraph says what reference-validation *does not* establish, in plain words, because leaving that silent invites the drift the Outsider flagged. Third, the mandatory dissent clause is load-bearing: any session that adopts reference-validation under protest from a participating perspective must record that protest, ensuring the methodology cannot silently reclassify objections as resolved.

What specifically changes about the receiving-session recording clause: under reference-validation there is no out-of-session completion path — the comparison is done in-session or hand-off to the next session, per Q3 — so the "may complete out-of-session" language does not apply. I leave it in for domain-validation's paragraph unchanged.

Does the "domain-actor who holds the problem" phrasing remain accurate? Yes, for domain validation. It does *not* remain accurate for reference validation, which is why I introduce a third paragraph rather than widening the second. The two validations are different objects and should read as different objects in the kernel.

## Q6. Iteration and pass criteria — the three-level question

**(a) Artefact pass.** A specific artefact passes reference-validation when: the comparison agent returns *structural match* or *defensible substantive divergence*; the contamination assessor returns *no-signal-of-contamination*; the within-session iteration count is at most one; and the constraint-counterfactual probe (per Q4(v)) shows the methodology produces a genuinely different artefact under a modified constraint. All four conditions must hold. If any fails, the artefact does not pass; if the contamination check fails specifically, the result is *no-signal*, not *fail*, and must be re-run with a different reference.

I note this is stricter than what the other perspectives will propose. That is deliberate. The looser the artefact-pass criterion, the more the mechanism approaches the ritual-validation failure mode.

**(b) Methodology pass.** I propose the following thresholds, higher than I expect Architect to propose. For the methodology's external-artefact claim to be considered validated at the methodology level: at least **five** reference-validation exercises completed; at least **four of five passing** under the artefact-pass criteria in (a); the five exercises must span at least three distinct domains (not all process-design, not all sequences); at least one must be in a domain where the selection step at Q1 criterion (3) placed the reference in the *difficult-to-contaminate* tail. If all five are in canonical-solution regimes per Q4's "no-defence-adequate" regime, none of them count toward methodology pass regardless of artefact-pass results.

Why five and not three. Reference validation has so much failure-mode surface area — contamination, selection bias, reference-quality variance, comparison-judge bias — that fewer than five samples gives almost no statistical purchase. Naming the pretrained input: this is a *small-n-inference* problem; three exercises give the methodology three opportunities to succeed at ritual without giving any opportunity to distinguish ritual from substance. Five is still small but at least allows a single failure without collapsing the claim.

Why four of five rather than five of five. A 100% pass rate on reference-validation would itself be a contamination signal. Real design activity on emergent-constraint problems should produce occasional defensible divergences that *don't* match the reference; a methodology that always matches the reference is more likely to be retrieving than designing. An 80% pass rate allows one "the methodology took a defensible but different path" result, which is the healthy outcome the mechanism should expect occasionally.

**(c) Mechanism-failure vs methodology-gap distinction.** This is the user's co-design steer and I want to engage it carefully because it is also where the mechanism is most vulnerable to the escape-hatch critique (see Q7).

A methodology gap, first-exercise: the methodology cannot absorb a specific category of emergent constraint (say, regulatory ones); the comparison reveals the gap; the gap is addressable by adding a kernel section or specifying an additional step; Session 016 records the gap and commits to addressing it in Session 017. This is a legitimate outcome and does not falsify the mechanism.

Mechanism failure, first-exercise: the mechanism *cannot return an interpretable pass/fail signal at all*. Examples: the comparison agent's findings are equally consistent with methodology-success and methodology-failure, repeatedly, across multiple re-runs; the contamination assessor fires on every run regardless of reference selection; the methodology consistently returns pass on contamination-present runs *and* pass on contamination-clean runs at the same rate, indicating the mechanism is not actually distinguishing; the judgment output is indistinguishable from random labelling under a held-out comparison.

Concrete threshold for declaring mechanism failure: if the first three exercises produce results where the pass/fail signal is not reproducible under rerun (same inputs, same agents, same seeds where applicable, different label), or where the contamination assessor and the comparison agent systematically disagree in ways neither can resolve, the mechanism is falsified and must be redesigned, not iterated.

The distinction between gap and failure is subtle and may not be cleanly resolvable in practice. That is my Q7 concern on this point: the user's generous framing of "failed first-exercise surfaces a gap" can absorb *any* negative result as a methodology gap rather than a mechanism failure. I propose the rule that, to be classified as a gap, the first exercise must produce a *specific, addressable* deficit — "the methodology lacks a step for handling regulatory constraints" is a gap; "the methodology's output didn't match the reference and the comparison agent couldn't say why" is not a gap, it is a mechanism failure in disguise.

## Q7. Risks and dissent — substantive engagement required

**Strongest failure mode.** I've named it throughout Q1–Q6: retrieval-masquerading-as-design on reference cases drawn from the canonical-solution regime, producing structural matches that are read as validation evidence but measure only the overlap of training distribution with reference distribution. Under my Q1–Q6 specification, this failure mode is *partially* defended: Q1(3) rules out the most obviously contaminated cases; Q4(v)'s counterfactual probe provides some positive evidence of design; Q6(a)'s contamination-dominates rule prevents obvious contamination from being called pass. The residual failure — sophisticated contamination on a carefully-selected case where the counterfactual probe happens to also be in-distribution — is the one I cannot defend against. I estimate it as the most likely outcome for any reference case meeting all Q1 criteria.

**Explicit falsifiability condition.** If Session 015's first exercise produces any of the following, the mechanism's adequacy is falsified and the methodology should return to pause:

- Produce-agent transcripts in which the agents identify the reference case by name before being shown the reference, or cite specific features of the reference's solution without derivation.
- Counterfactual-probe results where the methodology, given a constraint modified to be off-distribution for the reference, produces an artefact that still tracks the original reference rather than the modified constraint.
- Comparison results where the structural-match label was decided primarily on lexical overlap (e.g., the comparison agent quoted exact phrases as evidence of structural correspondence).
- Any result where the Produce agents or the comparison agent independently surfaced reference material the isolation procedure was meant to prevent access to.

I note the *undemonstrable-falsifiability problem* inherited from the Session 013 audit: my C4-placement falsifiability condition in Session 013 was "if a future domain-actor reports," and that condition is precisely what OI-016 flags as unavailable. The condition is unobservable under current methodology state, which makes that minority dissent unfalsifiable-in-practice. Reference-validation inherits the same structural risk here: the falsifiability conditions I just listed presuppose that Session 015 runs, that transcripts are produced, and that the workspace honestly surfaces contamination when detected. If any of those presuppositions fails — particularly the last, if the orchestrator has a motive to pass the exercise — the falsifiability conditions are inert. I name this rather than paper over it.

**Procedural self-deception test.** The concrete observable that would tell us the mechanism is ritual rather than substantive: the language the methodology uses about its validated artefacts drifts from "passed reference-validation under [specific conditions]" to "validated" simpliciter across sessions. If by Session 020 the workspace is referring to reference-validated artefacts with the same vocabulary used for domain-validated artefacts, the mechanism has laundered. A second observable: the Session N+1 kernel or spec-file edits that *widen* what counts as reference-validation pass (e.g., dropping the contamination check because "it kept firing," or raising the allowed iteration count because "the methodology needs more room"). Any such widening, without concurrent strengthening elsewhere, is evidence that the mechanism is being accommodated rather than evaluated. A third: absence of failures. If every reference-validation exercise passes, the mechanism is not measuring anything.

**Fallback-to-pause circumstances.** I propose the methodology returns to pause — i.e., OI-016 reopens and no new external artefacts are produced — under any of the following: the first exercise triggers any of the falsifiability conditions above; by the fifth exercise the methodology has not passed four; the procedural-self-deception observables are detected at any session; or the domain-actor situation changes (a domain-actor becomes available for a specific artefact, at which point that artefact should receive domain-validation and reference-validation should be retired for that artefact's class).

**The strongest case against Option (d) being adequate under any specification.** I inherit the Session 013 Skeptic's position at full strength and here is its Session 014 form.

The Session 009 revision named domain-validation as a distinct activity because workspace-validation alone was insufficient for artefacts that function outside the workspace. The insufficiency was *not* about the workspace's internal checking being weak; it was about the *kind of evidence* produced. Workspace-validation produces evidence that an artefact is internally consistent, structurally sound, and responsive to the constraints stated in-session. Domain-validation produces evidence of a different kind: that the artefact, deployed in the world, functioned as intended for someone who had the problem. These two evidence-kinds are not on a spectrum; they are different objects. The Session 009 insight was that the methodology had been conflating them, and the kernel revision made the conflation impossible going forward.

Reference-validation produces a third kind of evidence, and it is *closer in kind to workspace-validation than to domain-validation*. It evaluates whether the methodology's artefact is structurally consistent with a pre-existing artefact that is stipulated to be proven. It does not establish that the methodology's artefact would function for anyone in the world. The stipulation "this reference is proven" bridges the gap at the level of language but not at the level of evidence: the methodology's artefact inherits the reference's stipulated validity only if the methodology's artefact *is* the reference, and if it *is* the reference then the exercise measured retrieval, not design.

This is not a solvable problem through better mechanism design. It is structural. The mechanism Q1–Q6 specifies can be made as careful as anyone wants; it will still produce evidence of the third kind (structural correspondence with a pre-existing artefact), which is not the kind of evidence Session 009's kernel revision named as necessary. Naming the pretrained input for this: *ecological validity* from psychology — the question of whether a finding generalises from the test environment to the setting it claims to inform. Reference-validation has low ecological validity for claims about artefacts functioning in intended use; no mechanism refinement changes this.

Option (d) is therefore inadequate in principle to the problem OI-016 names. It can be adopted as a *less bad than nothing* substitute, and my Q5 revision to §7 reflects that framing explicitly ("provisional substitute").
