## D-051: Stress-test watchpoint findings (W1–W4) recorded per D-047.2; no mid-session spec revisions made

**Decision:**

All four D-047.2 watchpoints were probed during Session 008's Produce activity. Findings are recorded below. Each finding's disposition is either *genuine finding* (bit during Produce) or *mild finding* (noticed but did not bite hard). No mid-session spec revisions are proposed; per D-047.3, revisions are deferred to Session 009+ and require their own deliberation.

**W1 — Read activity for external domains. GENUINE FINDING.**

The kernel's Read activity (`methodology-kernel.md` §1) describes absorbing "the full current state of the workspace." For an external domain (movement / mobility), this wording does not translate cleanly. Session 008's Read activity for the external domain drew on:

- The user's stated constraints (passed in-session: stiff lower back, morning, mat available, no moves to rule out).
- General movement-design knowledge the orchestrating agent holds (gradual warmup principle; morning-tissue-state; supine→quadruped progression; mid-range before end-range; breath-led tempo; common physio/yoga vocabulary for lower-back sequences).

The kernel does not name the second sense — drawing on domain knowledge the agent or brief contains. It names only workspace-reading. A reader of the kernel applying it to an external domain would need to reinterpret Read to include domain-knowledge absorption.

This is a genuine gap. Session 009 or later may propose a kernel clarification adding a sentence to Read: "For work outside this workspace's self-hosting scope, Read also includes absorbing the domain constraints the session operates under and the domain knowledge available to the perspectives."

**W2 — Workspace structure for external work-products. GENUINE FINDING, with a tooling-interaction sub-finding.**

Sessions 001–007 produced methodology-internal artefacts (specifications in `specifications/`, tools in `tools/`, provenance in `provenance/`). Session 008 produces an external domain artefact with no pre-existing placement.

In-session resolution: the artefact lives at `provenance/008-first-external-application/artefact-morning-unfurl.md` — a **non-numbered filename** within the session's provenance directory. Numbered files (`00-assessment.md`, `01-brief-shared.md`, `01-deliberation.md`, `02-decisions.md`, `03-close.md`) retain the Session 001–007 convention.

The placement decision surfaced a tooling interaction worth recording as a sub-finding: an initial attempt placed the artefact at `02-artefact-morning-unfurl.md`, shifting decisions to `03-decisions.md` and close to `04-close.md`. `tools/validate.sh` (current v3 state) hard-codes `02-decisions.md` as the decisions-file path for checks 14 and 15 (trigger-coverage checks). Renaming decisions to `03-decisions.md` therefore silently bypassed checks 14 and 15 for Session 008's decisions — a structural-validation gap that would have been invisible to the failing-check report because the checks simply would not have run on `03-decisions.md`. Rather than modify `validate.sh` mid-session (which would be a mid-session tooling change inconsistent with D-047's spirit — the tool is also an artefact the methodology should observe pain on before revising), the artefact was renamed to a non-numbered filename and the methodology-internal files were returned to their standard `02-/03-` positions.

This is a **genuine finding with an actionable Session 009+ implication**: `validate.sh`'s hard-coded `02-decisions.md` path is brittle against legitimate session-structure evolution. A Session 009+ deliberation may propose either (a) a structural convention that external artefacts live somewhere that does not collide with the 00/01/02/03 numbered series (e.g., a new top-level `applications/` directory per Session 007 Q4's Generalist/Steward proposal; requires `workspace-structure.md` revision, D-023.2 trigger); or (b) a `validate.sh` update that accepts decisions-file at any numbered position, or that searches for `*-decisions.md`. Neither option is pursued in Session 008.

Alternative considered and deferred: creating a new top-level `applications/` directory in Session 008 (as Generalist and Steward proposed in Session 007 Q4, rejected pre-use per D-047). Creating `applications/` now would require revising `workspace-structure.md` (D-023.2 trigger → would require a non-Claude-participation deliberation). For one artefact in one session, in-provenance non-numbered placement is operationally adequate and avoids the spec-revision cost.

If future sessions produce multi-file artefacts, multi-session continuations, or multiple applications per session, the in-provenance placement will strain. The `applications/` directory question genuinely comes due on the second external application.

**W3 — Self-referential phrasing in active specifications. MILD FINDING.**

The kernel's activity names (Read, Assess, Convene, Deliberate, Decide, Produce, Validate, Record, Close) are abstract enough to translate to external domains without strain. The kernel's concrete language in §2 Assess ("Are existing specifications consistent with each other and with reality?") is self-referential — "specifications" here means methodology specs; for external work, the analogous question is "Are my domain-model assumptions consistent with what the user has told me?"

`validation-approach.md` (v3) is heavily self-referential: "Check that the workspace is internally consistent" — "workspace" means methodology workspace. For external-artefact Validate, this doesn't apply directly; the domain-native Validate is separately specified in D-046 criterion 3 and elaborated per artefact.

`multi-agent-deliberation.md` (v3) is also self-referential but the concepts (perspectives, convening, synthesis) translate — they're about deliberation structure, which is domain-agnostic.

During Session 008's Produce activity, I did not need to consult `validation-approach.md` or `multi-agent-deliberation.md`; they govern meta-level quality, not artefact content. Self-referential phrasing therefore did not bite hard. It would bite harder if a future session applied validation-approach's internal-consistency checks to an external artefact (a category error, but one the spec doesn't forbid).

Mild finding. Session 009+ may note the distinction between internal and external Validate explicitly in the validation-approach spec if a session needs both senses.

**W4 — Validate activity for external domains. GENUINE FINDING, partially resolved in-session.**

Two senses of Validate are in active use:

- *Internal Validate* — the kernel §7 sense: check that the workspace is internally consistent (specs don't contradict, provenance well-formed, open issues reflect state).
- *External Validate* — the domain-native sense: does the external artefact work as claimed in its domain? For Session 008's movement sequence: does the user attempt the sequence, does every move work, does it honour the constraints, does the low back feel different after?

Both senses are active in Session 008. Internal Validate is done in-session (validate.sh passes; see Validate section of close). External Validate happens OUT of session: the user tries the sequence the morning after and reports back at Session 009 open. Session 008 therefore closes with external Validate **pending**.

This is a genuine and load-bearing finding. The kernel's Validate wording is single-sense (internal). D-046 criterion 3 ("Validate analogue required") was the external-sense acknowledgment at target-selection time, but the kernel itself does not distinguish the two. Session 009+ may deliberate whether to revise the kernel to name both senses. Such a revision would be a D-023.1 trigger and would require non-Claude participation.

**Four watchpoints summary:** Two genuine findings (W1 Read; W4 Validate) warrant Session 009+ spec-revision deliberations. One genuine finding handled in-session (W2 Workspace structure — artefact placed within provenance; applications/ directory deferred). One mild finding (W3 self-referential phrasing — did not bite in Session 008's Produce; may bite later). No watchpoints are fixed in-session; findings are preserved for future sessions to act on per D-047.3.

**Rationale:** D-047.2 requires watchpoint findings be recorded; D-047.3 requires watchpoints be records, not revisions. This decision complies. The Skeptic+Outsider Session 007 cross-model position — "pain is the signal; removing the pain removes the learning" — is honoured: Session 008 observed and recorded where specs bit, rather than pre-emptively revising them. Session 009+ may act on the observations with its own deliberations.

**Rejected alternatives:**

- **Revise `methodology-kernel.md` mid-Session 008 to name external Read and external Validate.** Rejected because: (a) D-047.3 explicitly forbids mid-session watchpoint-fixes; (b) a kernel revision triggers D-023.1 and would require re-deliberation with non-Claude participation; (c) one session's observations are a single data point and may not justify a structural revision. Deferred to Session 009+.
- **Create `applications/` top-level directory now.** Rejected for Session 008's scope (D-023.2 trigger; workspace-structure.md revision; would require non-Claude-participation deliberation). In-provenance placement adequate for one artefact. Session 009+ may deliberate `applications/`.
- **Skip watchpoint recording on the grounds that Session 008 is "busy enough."** Not argued by any perspective but hypothetically available; rejected per D-047.2's recording requirement — watchpoints are mandatory session outputs, not optional.

**What remains open:**

- Whether Session 009+ revises kernel Read to name external-domain knowledge-absorption (W1 finding).
- Whether Session 009+ revises kernel Validate to name both senses (W4 finding).
