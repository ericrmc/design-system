## Q1. Which §8 remediation shape is right for MAD at 6,403 words?

**Position:** A.3 relocate-to-archive, with a surgically named seam: (a) the YAML field-block enumerations inside §Heterogeneous-Participant Recording Schema move to `archive/mad-schema/recording-schema-fields.md`; (b) the full §Open Extensions candidate-register moves to `archive/mad-schema/open-extensions.md`; (c) MAD retains the normative prose of both sections with `[archive-proposal: archive/mad-schema/...]` references at the points where detail was elided (using `archive-proposal:` token here to avoid validator check-22 false-positive on hypothetical-path text; under actual A.3 adoption the live references would use `[archive: path]`). Estimated residual MAD: 5,200–5,400 words. No file is split; no new default-read file is created.

**Reasoning:** The brief §3.1 content inventory is doing work the A.1/A.2 framing partially obscures. §Heterogeneous-Participant Recording Schema is 772 words, of which ~400 are YAML field blocks and code examples (brief §3.1). §Open Extensions is 671 words of `[activation precondition: ...]` items — the brief explicitly classifies this as a "non-normative futures register." Together that's 1,443 words of content that is either enumerative detail (the YAML) or explicitly-deferred futures. Removing them moves MAD from 6,403 → ~4,960; adding back ~100-word pointer paragraphs in each place puts residual at ~5,160, rounded to 5,200–5,400 for honest margin.

What distinguishes A.3 is the *shape of the content being moved*. A.1 compression would have to reach into normative prose to hit well-clear (<5.2K), because the large line-items are already the least-dense-per-word content; compressing them by 50% is still compression of schema detail, which §8's "must not summarise or silently compress" makes fragile. A.2 split along the OI-004 seam creates a new default-read file whose bulk (1,463 words) would be cleanly topical but doubles governance-text surfaces every session must read in full — not reducing the reading load. A.4 preserves the current arrangement, which is closest to correct in spirit but doesn't address that the bulk-heavy sections *already* invite skipping before 6K-soft-warn fired. A.3 is the only shape that says: governance text stays at default-read; enumerative detail moves where it can be reached by reference. The §Heterogeneous-Participant Recording Schema's own structure already invites this cut — the prose says *what must be recorded* and *why*; the YAML says *exactly how each field is typed*. Different readerships, different occasions.

I flag one disagreement with the brief §2 framing. The brief describes A.3's residual as "depends on what moves," which makes A.3 read as under-specified compared to A.1 and A.2's numbered residuals. My position commits to a specific number and specific files so A.3 can be evaluated on the same footing.

## Q4. If the chosen shape relocates, how does it engage WX-22-1 laundering-as-codification?

**Position:** A.3 as I have specified it *repairs* the read-contract's intent rather than *codifying* the laundering pattern, but the repair-vs-codification test turns on precisely which content moves and what stays. My A.3 moves YAML field-block enumerations and a non-normative futures register — content that is already treated as reference material by the structure of the existing section. It does *not* move normative prose describing what must be recorded, why the three-layer schema exists, or what the heterogeneous-participant requirements are. The test is: does the default-read text, read in full, give a session everything it needs to *run* the deliberation discipline correctly? Under my A.3, yes. Under a broader A.3 that relocated more normative prose, no.

**Reasoning.** WX-22-1's concern (brief §4.6) is that archive-pack reference could codify the harness-layer routing-around-oversized-files pattern OI-015 named, dressed in read-contract clothing. My A.3 rejects that. The YAML field blocks in §Heterogeneous-Participant Recording Schema are *not* governance text — they're an enumeration of field types and value formats a session consults when writing a recording entry ("is this field required? what's the value schema?"). The normative prose surrounding the YAML — "the recording schema has three layers"; "trigger annotations are required for every deliberation"; "agent-kind-manifest must be recorded" — is what every session needs at open to know the discipline. That prose stays at default-read, unchanged, with a pointer where the YAML blocks used to sit.

Same logic for §Open Extensions. Brief §3.1 classifies this 671-word section as "non-normative futures register"; items are flagged `[activation precondition: ...]` — they announce themselves as not-yet-live. A session reading MAD at open does not need the full candidate-register to run its deliberation; it needs to know the register exists, where to find it, and what category of content it contains. An ~80–100-word pointer carries that at default-read; the register lives in `archive/mad-schema/open-extensions.md` where a session activating a candidate (or a synthesis surveying the field) consults it by reference.

The Session 025 operator-discipline test: would Session 025 actually read only default-read MAD at open *and* reach for `archive/mad-schema/open-extensions.md` only when an item becomes relevant? I think yes, for the same reason Session 025 wouldn't read `decisions/index.md` entries from Sessions 001–019 at open — default-read is for what's needed to run the discipline; detail accessed by reference is accessed when the reference is needed. The `validation-approach.md` v5 discipline catches a session that *should* have consulted the archive and didn't: referencing a field whose schema lives in archive requires consulting the archive to quote it correctly. The archive isn't dead-storage; it's reference material with a live-access path.

**What A.3 does worse than A.1 / A.2 / A.4.** A.3 introduces a default-read-plus-archive pattern operators must internalise — at each `[archive-token: path]` the reader must decide *now* whether the archived detail is needed. A.1 (everything in one file, no decision) and A.4 (same as A.1) don't impose that. A.2 imposes a different cost (two default-read files at open) but the decision point is at file-open rather than mid-read. A.3's cost: operators must trust the classification and have a reliable archive-access path. If archive access is brittle (path drift, index drift), A.3 fails where A.1/A.2/A.4 don't. I think this cost is worth paying because the alternative — carrying enumerative detail at default-read forever — is worse. But the cost is not zero.

**Some readers will miss detail they'd have caught at default-read.** Concretely: a session opening MAD v5 that doesn't consult `archive/mad-schema/recording-schema-fields.md` won't see the full YAML for `participant.agent_kind_manifest` at open. If that session's deliberation then requires writing an agent-kind-manifest, the operator must reach for the archive — friction A.1/A.4 don't have. I accept that cost because (1) sessions not needing the YAML currently pay to read 400 words of it at every open for nothing; (2) sessions needing it must write the field correctly regardless, which means reaching for authoritative schema regardless — having it in a named archive is not worse than having it buried in MAD. The repair: default-read gives governance; archive gives schema detail; both preserved verbatim; neither summarised.
