understanding of it, particularly how the language is casual and friendly." The user
explicitly confirms that the artefact meets the criteria the user named in the Session 010

governance steps or approvals. So it meets the criteria there." The user also reports
that "the steps make sense and could be followed easily."

This confirms:

- **The overall five-move structure functions as designed.** The user's critique is
  local to specific moves and a specific example line, not structural.
- **The register discipline (per D-057.3, non-corporate, non-therapy) held in
  domain use.** The user's only register complaint is the single example line in
  Move 1 ("Shelving for the lounge, around $500...") — see below.
- **The "allowed flexibility for resolving the problem" property** that Session 010's
  synthesis aimed for (pause permission between Move 3 and Move 4; Outsider-originated
  asynchronous/fragmented-use structural requirement per WX-10-2) was received as

This is positive Validate evidence for Session 010's primary claim. It is not strong
evidence for the methodology's domain-general claim (still n=2 with WX-10-4 under-power
concerns) — but for the Session 010 artefact's own success, this is a pass.

### Specific local corrections

The user requests four concrete changes:

1. **Move 1 "Name the decision" is too robotic and names too many constraints up front.**
   The user's own language for how such a discussion starts is quoted: *"Should we talk
   about the lounge?"* or *"What should we do about the lounge?"* — these are questions
   and invitations, not declarative constraint-laden sentences. Additionally: *"we
   don't know all the constraints up front; these might only surface when we try to
   find a balance between what someone cares about and how much there is to spend, or
   what we need to defer and save for."* The current Move 1's example line (*"Shelving
   for the lounge, around $500, before the end of the month."*) presumes scope, budget,
   and timeline all known at the opening — which the user reports is not how these
   conversations actually begin in this household.

2. **The example line in Move 1 reads corporate.** The user's only register complaint:
   *"except the first suggested line 'Shelving for the lounge, around $500...'"*. The
   rest of the artefact's register is endorsed; this one line is flagged as an
   exception. Synchronises with correction (1): a question-shaped opening line would
   also be register-consistent.

3. **Move 5 must name the person responsible for following up.** The current Move 5
   closes the decision but does not name who owns the follow-through. The user reports
   that follow-through has "some mental burden on that person as they need to schedule
   it, order it, decide how to receive it (delivery or pick up, making sure someone is
   available)." Owning the decision is not the same as owning the execution; the
   artefact currently treats them as one.

4. **Closing out often includes handling older items being replaced.** For furniture
   specifically, the user names a sub-sequence: *"We have to assess if it can be
   reused, or donated, or work out how to dispose of it."* This is an artefact-local
   domain finding (furniture-domain-specific rather than household-decision-general);
   the user's own framing is that "a few intricacies which vary by domain" were left
   out and "in this example of buying furniture it required more steps to close out."

Corrections 1 and 2 are closely linked (both concern Move 1's opening). Corrections 3
and 4 are closely linked (both concern what happens after the decision is made). The
session deliberates how to incorporate each — see `01-brief-shared.md` and
`02-decisions.md`.

## Methodological Significance

Three specific properties of the user's report are methodologically informative beyond
the Validate outcome itself:

1. **The "few intricacies which vary by domain" framing is domain-generality data.**
   The user's explicit observation that some intricacies are furniture-specific
   (disposal/donation of old furniture is named) and others are general (naming the
   person responsible for follow-up applies to any domestic decision) is *domain-
   typing data* the methodology has not previously surfaced. Session 010's artefact
   was produced under a brief that assumed the protocol would be domain-agnostic
   within household-decision scope; the user's feedback shows that the household-
   decision scope contains sub-domains (furniture, landscaping, appliances,
   renovations) with different closeout intricacies. Session 013's revision must
   decide which corrections are artefact-general versus which are artefact-specific.

2. **The user's corrections include both language-level and content-level changes.**
   Move 1 and the example line are *language* corrections; the Move 5 additions are
   *content* corrections (new steps, not rephrased steps). The artefact's revision
   therefore touches two distinct layers — and the Skeptic's Session 010 dissent
   (preserved in D-057.9 and D-058's WX-10-3) specifically predicted that governance-
   domain artefacts may need revision in use. The user's pattern is consistent with
   that dissent.

3. **The standing-constraint statement changes the methodology's operational
   context, not its specifications.** No specification is wrong; the methodology's
   external-artefact production path remains as specified. But a key input to that
   path — the user's availability as domain-actor — is withdrawn for future work.
   Session 013 should not rush to patch the specifications in response; the right
   response is to name the new open issue, record its activation triggers, and let
   a future session deliberate whether specification changes are warranted when
   the issue becomes active.
