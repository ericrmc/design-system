## Q1 — Yes: the workspace should adopt full reframing (H2) because the engine already exists de facto and naming it is truth-surfacing, not speculative design

I argue for H2. The workspace has accumulated sixteen sessions of structural ambiguity in a single word — "methodology" — that carries two loads simultaneously: an abstract-approach load (the mechanic of diverse perspectives producing durable artefacts) and a concrete-implementation load (this specific workspace's files). Session 016 surveyed this and found it latent throughout the six active specifications. That is not a terminology quibble; it is a load-bearing ambiguity that shapes how every new session orients itself and how every specification is read. The strongest reasoning for H2 is that the engine is already separable — `PROMPT.md` plus `methodology-kernel.md` plus `multi-agent-deliberation.md` plus `validation-approach.md` plus `reference-validation.md` plus `identity.md` plus `tools/validate.sh` constitute a coherent loadable unit that can execute against any input, and this has been true since roughly Session 010. Naming this unit is a truth-surfacing act that makes the workspace legible to itself and to any eventual outside reader. The main cost is revision burden: H2 touches five specifications, restructures `PROMPT.md`, adds a new manifest, and forces a decision on `identity.md`'s scope. That is real work, and some of it will be retouched when an actual external application teaches us what we got wrong. I accept that cost because the ambiguity compounds — every future specification written under the current framing deepens the two-sense conflation, and the cost of retrofitting grows monotonically with workspace size.

## Q2 — Five specifications change; the shape is concrete per file

`PROMPT.md` is reorganised (see Q3). `methodology-kernel.md` v4 is retitled to `engine-kernel.md` v5; the nine-activity execution semantics are unchanged in substance, but every occurrence of "the methodology" as concrete-implementation is rewritten to "the engine," and the §1 framing paragraph explicitly states: "This specification defines the execution kernel of the Selvedge engine. It is part of the engine's loadable definition, distinct from the development provenance that has produced it." The §7 three senses of Validate remain as written. `workspace-structure.md` v2 becomes v3 with a new §-level distinction between three file classes: **engine-definition files** (the loadable unit — `engine-kernel.md`, `multi-agent-deliberation.md`, `validation-approach.md`, `reference-validation.md`, `identity.md`, `tools/validate.sh`, and an engine-prompt), **development-provenance files** (`SESSION-LOG.md`, `provenance/`, `open-issues.md`, and the development-prompt), and **application-scope files** (`applications/*/`). The `applications/` description is extended: "Each application directory contains the application's own scope, inputs, outputs, and — if the application uses external-problem framing — its own application-prompt that loads the engine definition plus the application's context." `identity.md` v1 becomes v2: "Selvedge names **the engine** — the loadable unit defined by the engine-definition files at any engine version. The methodology as abstract-approach is not separately named; it is whatever Selvedge instantiates. The workspace as development context is also not separately named; it is the provenance of Selvedge's own development." The four Reopening conditions are preserved verbatim. `reference-validation.md` v1 is unchanged in substance; its §1 adds one sentence acknowledging that reference-validation is an engine-level activity that applies uniformly across all applications of the engine. `multi-agent-deliberation.md` v3 and `validation-approach.md` v3 are unchanged in substance; both get a single-sentence header clarifying they are engine-definition specifications. The new file is `engine-manifest.md` v1 (Q4). Five specification files touched; one new file; one rename.

## Q3 — PROMPT.md splits into three files along already-latent seams

I propose a split, not a branched single file. Branched single files create their own cognitive cost (readers must track which branch applies) and the current `PROMPT.md` is already structured such that the split is cheap. Concretely: `engine-prompt.md` contains the engine-level operating content — the "How to operate" sections, the nine-activity discipline, the multi-perspective deliberation requirements, the validation discipline, the anti-silent-import discipline, and the "Rules that hold across applications." It does **not** contain the self-development-specific framing. It ends with a placeholder: "The application context follows below or in a companion file. Begin by Reading the application's initial state." `development-prompt.md` loads `engine-prompt.md` by reference (one line: "The engine prompt at `engine-prompt.md` governs execution; what follows is the application-specific scope for this workspace's foundational application: Selvedge's own development") and then contains the self-development-specific content — the "What this methodology is" framing rewritten as "This application is the engine's own development; each session evolves the engine's own specifications by running the engine on its own outputs," the `SESSION-LOG.md` orientation, the open-issues orientation, and the "Now begin" entry point. `application-prompt-template.md` is a third file sketching the pattern for external-problem applications: it loads `engine-prompt.md` by reference, then provides named slots for application scope, inputs, success criteria, and initial state. This is a template, not a live prompt. The rationale: the self-development-specific framing is concentrated enough in the existing `PROMPT.md` that the split is mechanical rather than interpretive. The bootstrap entry point for the current workspace becomes `development-prompt.md`; the entry point for any future external application is a copy of `application-prompt-template.md` filled in.

## Q4 — Yes: engine-manifest.md is low-overhead and high-leverage

`engine-manifest.md` v1 is a short specification (I estimate 2–3 KB) with the following section outline: **§1 Purpose** — names the engine as Selvedge and states that this manifest enumerates the files constituting the engine at the current engine version. **§2 Current engine version** — a version string (proposal: `selvedge-engine-v1`; this is the first version under the reframing). **§3 Engine-definition files** — enumerated list: `engine-prompt.md`, `engine-kernel.md`, `multi-agent-deliberation.md`, `validation-approach.md`, `reference-validation.md`, `identity.md`, `tools/validate.sh`. Each entry names the file and its role in one sentence. **§4 What is explicitly not part of the engine** — `SESSION-LOG.md`, `provenance/`, `open-issues.md`, `development-prompt.md`, `applications/*/`, this manifest's own version history. **§5 Versioning discipline** — an engine version changes when any file in §3 changes in substance (not in typography). Kernel revisions bump engine version; adding a new engine-definition specification bumps engine version; changing the engine-prompt's operating semantics bumps engine version. **§6 Loading the engine** — a one-paragraph description of what "loading the engine" means operationally: reading §3's files in order, treating them as a closed unit, and then consulting whatever application-prompt specifies the application's scope. The leverage of this manifest is that it makes the engine pointable-at. Kernel v4 → v5 becomes `selvedge-engine-v1.0 → v1.1` traceable in the manifest's version history rather than requiring archaeological reconstruction from session logs. Any future external application can state "uses selvedge-engine-v1.x" and that is unambiguous.

## Q5 — Selvedge names the engine; H2 likely triggers Reopening condition 2

Under H2, Selvedge names **the engine** — the loadable unit defined by `engine-manifest.md`. It does not name the methodology in the abstract-approach sense, and it does not name the workspace. This is the cleanest resolution of the two-sense conflation: the abstract-approach sense is not a thing that needs a proper name (it is a category, not an instance), and the workspace is a development context, not a product. On Reopening conditions: condition 1 (external adoption threshold) is unmet and H2 does not change its status. Condition 2 (semantic drift — the name becoming misleading) is plausibly triggered by H2 itself: if `identity.md` v1 said Selvedge names the methodology, then redefining what Selvedge names is a semantic-scope change. I argue this is a Reopening that immediately closes with the v2 revision, not a reason to avoid H2. The act of clarifying scope is itself the Reopening response. Conditions 3 (methodology-renaming pressure from operator feedback) and 4 (confusion reports from external users) are unaffected. [uncertain] Whether condition 2 is formally triggered versus whether H2's adoption is simply a scope-clarification within the existing name is a judgement call the Decide activity of this session must make.

## Q6 — External-application activation path is: copy template, fill slots, execute

Concretely, a new user who wants to apply Selvedge to their own problem does the following. They copy `engine-prompt.md`, `engine-kernel.md`, `multi-agent-deliberation.md`, `validation-approach.md`, `reference-validation.md`, `identity.md`, and `tools/validate.sh` into a new directory as-is (or clone the engine-definition from a published location — out of scope here but enabled by the manifest). They copy `application-prompt-template.md` and rename it to `PROMPT.md` in their workspace. They fill in the template's application-scope slots: problem statement, inputs, domain-validation approach, success criteria, initial state. They create an empty `SESSION-LOG.md`, an empty `open-issues.md`, an empty `provenance/`, and an empty `applications/` (if their workspace will produce artefacts). They execute the filled-in `PROMPT.md`, which loads the engine via the engine-prompt reference and then runs the engine against their application scope. The first session Convenes perspectives, Deliberates the problem's opening structure, Decides an initial direction, Produces whatever the application's success criteria call for. The engine's execution semantics apply uniformly; only the application-specific scope differs. This is the concrete answer to "how does someone else load Selvedge": they load the engine-definition files, they fill a template, they run. Without H2, this path requires reverse-engineering the boundaries from the existing `PROMPT.md` and guessing which files are engine versus workspace-specific. With H2, the boundaries are explicit and the template is a real artefact.

## Q7 — Strongest counter-argument: H3 captures 80% of H2's value at 20% of the cost, and the observable condition that would flip me is first-external-application friction

The strongest argument against H2 is that H3 — new `engine-manifest.md`, minor updates to `identity.md` and `PROMPT.md`, no split of `PROMPT.md`, no rename of existing files — captures the truth-surfacing act (naming the engine as a distinct loadable unit) at a fraction of the revision cost. Under H3, the manifest enumerates the engine-definition files; `identity.md` is updated to say Selvedge names the engine; `PROMPT.md` gets a single paragraph acknowledging the two application kinds. The split into `engine-prompt.md` and `development-prompt.md` does not happen until an actual external application needs it. This is the Skeptic's premature-structural-commitment argument in its strongest form: Session 012 refused to name the methodology until it had earned the name; by the same reasoning, splitting `PROMPT.md` before a real external application exists imports a structural commitment the engine has not yet earned. My response is that naming the two-sense ambiguity costs zero evidence (the ambiguity is demonstrable from Session 016's survey), but splitting `PROMPT.md` does cost evidence. The observable condition that would flip me from H2 to H3 is this: if the first real external application, under the H2 split, reveals that `application-prompt-template.md`'s slots were wrong — that real applications do not fit the template's shape — then H2's pre-specification cost was wasted and H3's wait-for-evidence discipline was correct. The symmetric condition that would confirm H2 is the first external application loading cleanly by filling the template's slots without structural modification. Until the first external application runs, this is genuinely uncertain; I argue for H2 because the ambiguity cost is accumulating now and the split's cost is bounded and recoverable. But I hold the position loosely: if H2 is adopted and the first external application reveals the split was premature, the engine should merge `engine-prompt.md` and `development-prompt.md` back into a single branched file at that point, treating the failed split as evidence rather than sunk cost.
