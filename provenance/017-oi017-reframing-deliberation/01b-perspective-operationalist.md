## Q1 — Adopt H3: name the engine via a new manifest, defer the rest until operational pressure arrives

I argue the workspace should adopt a reframing, but only the smallest increment that names the distinction Session 016 surfaced. Concretely: create `specifications/engine-manifest.md` v1 as the first artefact that enumerates which files constitute the Selvedge engine at any given version; leave `PROMPT.md` un-split; leave `methodology-kernel.md` un-renamed; touch `identity.md` with a one- or two-sentence clarification that Selvedge names the methodology and the manifest names its concrete engine-form.

My strongest reasoning is that this is the only move among the three that is *supportable by the evidence the workspace currently holds*. Sixteen sessions of self-development and two external artefacts (`morning-unfurl.md`, `house-decision-five-moves.md`) generated by the operator running the engine himself — this is not a corpus from which to infer the right shape of an external-application execution prompt. H2 specifies that shape now. H3 names the distinction now and lets the shape be specified when an external application actually launches against it.

The main cost of H3 is that it is a *weaker* response to the Session 016 observation than H2. H3 does not fix the "This workspace is building a design methodology" opening of `PROMPT.md`; it does not relieve a hypothetical external user from reading provenance they do not need; it does not, on its own, make the engine loadable as a unit in any executable sense. A new user under H3 has to read the manifest, then cross-reference the files it names, and mentally reconstruct what is engine-definitional versus development-provenance. That is a real cost and I accept it as the price of not pre-committing.

## Q2 — File-by-file changes under H3

Three files change. No files are renamed. No files are split. No files are superseded.

1. **New: `specifications/engine-manifest.md` v1.** Created as a fresh v1 spec. Its contents are drafted in Q4. No dependency on other specifications being revised simultaneously.
2. **`specifications/identity.md` minor revision (v1 to v2).** One- or two-sentence insertion. Proposed language: *"Selvedge names the methodology, which is concretely implemented as an engine. `engine-manifest.md` enumerates the files that constitute the engine at any given version. When this document says 'Selvedge,' it refers to both senses unless the context distinguishes them."* This preserves the current identity statement (Selvedge, from the textile term) and adds the two-sense clarification without forcing the identity decision into either/or form.
3. **`PROMPT.md` one-paragraph addition (no version bump unless the methodology's conventions require one).** The addition points at `engine-manifest.md` and acknowledges that the current `PROMPT.md` drives the self-development application. Proposed insertion point: near the top, after the first framing paragraph. Proposed language: *"This `PROMPT.md` is the execution prompt for the self-development application of Selvedge. The engine itself is enumerated in `specifications/engine-manifest.md`. Applications of Selvedge to external problems are expected to use their own execution prompts that load the engine definition plus the application's own context; this file is not such a prompt."*

What does NOT change under H3: `methodology-kernel.md` (not renamed, not revised); `workspace-structure.md` (the `applications/` directory already exists from Session 009; the top-level structure needs no revision for H3); `multi-agent-deliberation.md`; `validation-approach.md`; `reference-validation.md`. The change footprint is deliberately narrow.

## Q3 — `PROMPT.md` stays single; no internal branching

I argue `PROMPT.md` does not need to be split, and does not need internal branches, at this step. Three reasons.

First, the second execution surface — the external-application prompt — currently has zero users. Designing the split now means designing against a hypothetical. The workspace has a pattern it invokes in its own discipline (the W-watchpoint concept; the Session 007 "external application preparation" work shape I understand was about shaping for external use *when external use becomes imminent*, not speculatively): design from operational pressure, not speculation. Splitting `PROMPT.md` before any external application has been launched is precisely the kind of speculative design that pattern resists.

Second, `PROMPT.md` read in full is not onerous for a future external user. It is 9 KB. If a user under H3 wants to apply Selvedge to their own problem, the manifest tells them which files constitute the engine; they can read `PROMPT.md` and mentally substitute their own application context. The friction this imposes is a one-time cost per external user; the friction of maintaining a split-file structure that has not been validated against real external use would be ongoing.

Third — and this is where I grant real weight to the opposing position — the "This workspace is building a design methodology" opening phrase *is* a friction point. But a one-paragraph addition (Q2 item 3) pointing at the manifest and acknowledging that this `PROMPT.md` drives the self-development application addresses the friction without committing to an untested split.

If the first external application is launched and its operator reports that the split is genuinely necessary, H2's restructure becomes available as a follow-on. H3 does not foreclose H2; H3 sequences it.

## Q4 — `engine-manifest.md` section outline

The manifest is the highest-leverage artefact H3 produces. Its minimum viable content is an enumeration. Draft section outline:

1. **Purpose.** One paragraph: the manifest names which files, at any given engine version, constitute the loadable Selvedge engine as distinct from its development provenance.
2. **Current engine version.** A version identifier (e.g., `engine-v1`) that increments when the manifest's enumeration changes. Not the same as any individual spec's v-bump.
3. **Engine-definitional files at current version.** The list. At v1, I expect this to be: `PROMPT.md`, `specifications/methodology-kernel.md`, `specifications/multi-agent-deliberation.md`, `specifications/validation-approach.md`, `specifications/reference-validation.md`, `specifications/workspace-structure.md`, `specifications/identity.md`. [uncertain whether `identity.md` belongs in the engine set or in a neighbouring "engine-identity" set — the deliberation may want to examine this.]
4. **Non-engine files in the workspace.** Provenance (`provenance/`, `SESSION-LOG.md`, `open-issues.md`), applications (`applications/`), tooling (`tools/`). Named so the distinction is explicit.
5. **Versioning discipline.** When a listed file's version changes in a way that changes engine behaviour, the manifest's engine version increments. When a listed file's version changes without behaviour change (e.g., typo fix), the manifest is unchanged.
6. **Invariants.** Short list of what the engine guarantees across versions: nine-activity execution semantics from `methodology-kernel.md` §X; multi-perspective deliberation when D-conditions are met; two-tier validation with reference-validation as a third sense; self-hosting property (the engine can be run on its own specifications). [uncertain how tightly to specify; I would err toward sparse at v1 and let pressure add specificity.]
7. **Relationship to applications.** One paragraph: applications of the engine consist of the engine-definitional files plus an application-specific execution prompt plus application-specific context. The self-development application is one such application, using `PROMPT.md` as its execution prompt. External applications would use their own.

The manifest is a spec about specs. It does not duplicate their content; it enumerates and versions them.

## Q6 — Activation path for a new external user

Under H3, a person outside the operator who wants to apply Selvedge to their own problem:

1. Clones or references the workspace.
2. Reads `specifications/engine-manifest.md` first — this is the legible entry point the manifest creates.
3. From the manifest's enumeration, reads the engine-definitional files (listed in Q4 item 3). Skips provenance and applications unless curious.
4. Reads `PROMPT.md` and notes the one-paragraph addition (Q2 item 3) stating that `PROMPT.md` is the self-development execution prompt.
5. Writes their own execution prompt for their application, modelled on `PROMPT.md` but swapping the self-development framing for their own problem context. Alternatively, copies `PROMPT.md` and edits the opening paragraphs. [uncertain which pattern the first external user will prefer; H3 is deliberately neutral between them.]
6. Creates a new workspace (or subdirectory under `applications/`) containing their execution prompt plus their own context files, and runs sessions against it using the manifest-enumerated engine files as the loaded engine.

The friction in step 5 is real and is H3's main cost (see Q1). The benefit is that the first external user's experience of step 5 tells us what `PROMPT.md`'s split should actually look like, at which point H2 becomes an informed restructure rather than a speculative one.

## Q7 — Strongest argument against H3 and the condition that would move me to H2

The strongest argument against H3 is the **cosmetic-reframing charge**: H3 names the engine-vs-methodology distinction but does not act on it where the action matters. A manifest that points at files without restructuring the files it points at is arguably underpowered — it gives a future external user a legible entry point but does not relieve them of the friction that `PROMPT.md`'s self-development framing creates. If that friction is the actual barrier to external adoption (and it plausibly is — the "This workspace is building a design methodology" opening is not neutral), then H3 addresses the Session 016 observation in form but not in function, and H2 is the only response that addresses both.

A related charge: if H2 is the eventual destination, H3 first is wasted motion. The manifest built under H3 will need to be revised when H2 happens (file renames, the split `PROMPT.md`, likely a new engine-kernel document). H2-now avoids the double revision.

I find the cosmetic-reframing charge strongest. I would not find it strong enough to abandon H3 today because the workspace has no operational evidence that `PROMPT.md`'s friction is actually blocking anyone — there is no anyone yet. But that is precisely the observable condition.

**The condition that would move me from H3 to H2:** the first external user (named practitioner, not the operator) attempts to apply Selvedge to their own problem under H3, and reports that `PROMPT.md`'s self-development framing is a concrete barrier they had to work around, not merely a minor reading cost. If that report arrives, H2's split becomes operationally justified and I would support the restructure. Until that report exists, H2 is designing against a hypothesis.

A weaker condition that would move me to H1 (no reframing): the manifest, when drafted, turns out to enumerate essentially the entire `specifications/` directory plus `PROMPT.md`, such that the "engine" set and the "specifications" set are nearly identical. In that case the manifest adds taxonomic noise without information, and recording the Session 016 observation as prose in an existing file (H1) is the more honest move.
