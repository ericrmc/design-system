## 2. Problem statement

S133 closed structurally without a `close_record` row. The proximate cause: a Python script ran `bin/selvedge submit close-record` followed by `bin/selvedge submit session-close` sequentially, without checking the close-record return code. close-record refused (`items[2]` atom was 289 characters, exceeding the 240-char atom limit). session-close then succeeded — no T-trigger checked for the absence of a close_record row.

Substrate state confirms: S133 is `status='closed'`, `closed_at='2026-04-30T00:08:05.182Z'`, with zero rows in `close_records` referencing its `session_id`. Recovery via post-close calibration-EF was itself blocked because `engine_feedback` requires an open session. The provenance gap is reconstructable from DV-S133-1 + EF-S131-1 + EF-S132-1 + EF-S133-1 + OI-S133-1 + commit messages — but the canonical close artefact is missing.

**The question this deliberation answers**: which substrate defences should ship to prevent recurrence, and at what enforcement layer?

The operator named two candidates:
- **Defence A**: session-close handler refuses if no close_record row exists for the session (`E_SESSION_CLOSE_WITHOUT_RECORD`-style refusal).
- **Defence B**: atom-length pre-check at payload deserialisation in the Python CLI, surfacing length violations at parse time rather than as `CHECK` refusals at INSERT.

The question is not *whether* to act — the gap is real and the operator has already directed remediation. The question is *what shape*: which defences ship, where in the enforcement stack they live, and what is left to operator policing.
