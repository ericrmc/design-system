# OI-085-001 — re-decompose v2 spec_versions

## Why

S085 registered v2 rows for `prompt-development` and `prompt-application` after S084 edited the prompt markdown bodies, clearing the validator drift on `body_sha256`. The v2 rows ship with `body_canonical_in_substrate=0` because no spec_sections / spec_clauses were authored against them — the SHA bump was a pointer-only registration. The same condition holds for `engine-manifest` v20.

S086's substrate-invariant audit confirms the gap: `engine-manifest-v20`, `prompt-application-v2`, and `prompt-development-v2` are `active` but each has 0 sections and 0 clauses. The dispatcher's active-spec query (`PROMPT.md` §Read the active engine spec) returns empty rows for these specs and silently falls through to reading the markdown bodies — exactly the failure mode OI-085-002 names.

## What's needed

A spec-decomposition subagent run, distinct from the orchestrator, that:

1. Reads each markdown body from disk (`prompts/development.md`, `prompts/application.md`, `specifications/engine-manifest.md`).
2. Submits `spec-section` rows for each heading and `spec-clause` rows for each clause, against the v2 spec_version rows already in place.
3. Flips `body_canonical_in_substrate=1` on each v2 row once decomposition is complete.

The v1 markdown structures are already captured (40 + 38 + 30 clauses for engine-manifest-v19, prompt-application-v1, prompt-development-v1). The v2 bodies are similar in shape; cost is proportional to the diff between v1 and v2.
