# Substrate enforcement of the coding review loop

Coding review loop is operator-policed; substrate does not refuse a coding session that closes without a clean reviewer pass.

S083 D-2 (substrate D-S004-2) introduces the coding review loop: any session producing, modifying, or deleting executable logic must invoke a reviewer subagent, address every medium-or-higher finding, and repeat until clean. The mechanism is currently operator-policed; the substrate does not yet refuse a coding session that closes without a clean reviewer pass or an explicit halted-state record. Why it matters: honest tool use does not survive context pressure across sessions (constraints.md §2). The next session writing executable logic could quietly skip the loop, or fail to record findings, with no substrate-level signal. What's needed: a substrate constraint that ties session-close on coding sessions to evidence of a clean (or explicitly-halted) reviewer pass — likely a new review-pass record table referenced from session-close, gated by a coding-session predicate. Deferred to post-release-gate; not load-bearing on the first 30-session external-problem trial.
