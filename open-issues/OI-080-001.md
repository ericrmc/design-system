### OI-080-001: T-06 missing trigger on `deliberations` UPDATE; T-13 admits non-NULL→other-non-NULL on sealed_at

**Surfaced:** Session 080 reviewer-agent pass 1 (HIGH-1 + MEDIUM finding).

**Status:** Open — substrate hole acknowledged; remediation requires migration 002 which depends on the `selvedge migrate` runner deferred per `engine-feedback/inbox/EF-079-002-T15-deferred.md`.

## The hole, in two parts

### Part (a) — T-06 absent on deliberations UPDATE/DELETE

The T-06 trigger family (`state/migrations/001-initial.sql`:405–570) covers post-close immutability for `decisions`, `decision_alternatives`, `spec_versions`, `perspectives`, `synthesis_points`, `commitments`, and `refs` — but **not** the `deliberations` row itself. A closed-session deliberation's `topic` and `synthesis_md` columns can be mutated via direct SQL UPDATE without the trigger firing. The CLI does not expose a path to mutate these columns post-seal, but any out-of-band writer (debugging script, migration that goes wrong, future writer-role with capability) can silently rewrite a closed deliberation's topic or its post-hoc synthesis text. This breaks the methodology's preservation guarantee that closed-session content is immutable.

Demonstrated by `state/tests/test_deliberation_kinds.py::test_t06_deliberations_update_unprotected_xfail` (xfail, strict). The test attempts `UPDATE deliberations SET topic='mutated' WHERE deliberation_id=?` against a closed session and expects `E_REFUSAL_T06`; today the UPDATE succeeds. When migration 002 lands the missing trigger, the xfail flips to xpassed and the test must be unmarked.

### Part (b) — T-13 only refuses non-NULL→NULL transitions on sealed_at

`state/migrations/001-initial.sql`:152–158 — T-13 fires `WHEN OLD.sealed_at IS NOT NULL AND NEW.sealed_at IS NULL`. It does not refuse `OLD.sealed_at IS NOT NULL AND NEW.sealed_at IS <some other timestamp>`. A direct SQL `UPDATE deliberations SET sealed_at='2025-01-01T00:00:00.000Z' WHERE …` against a sealed deliberation succeeds, retroactively rewriting the seal time. T-05 (no perspective insert when sealed) is enforced from the post-update value, so a retroactive re-seal does not unlock perspective insertion — but it does pollute the audit trail.

Demonstrated by `state/tests/test_deliberation_kinds.py::test_t13_permits_resealing_to_other_timestamp_xfail` (xfail, strict).

## Why it is not fixed in S080

Adding triggers requires either:
- Editing `state/migrations/001-initial.sql` in place — invalidates the sha256 stored in `schema_migrations` and violates the workspace's preservation rule (specs evolve via new versions, not silent edits). Refused.
- Authoring `state/migrations/002-…sql` and a `selvedge migrate` runner that applies it idempotently against existing substrates. The runner is the deferral recorded as `engine-feedback/inbox/EF-079-002-T15-deferred.md`. Building it in S080 would break the Option-C scope cap chosen at the start of this session (deliberator-CLI-extension only) and re-violate the 078 D-5 release gate's "no methodology-expanding self-development" intent if it touched the workspace's migration discipline rather than just the runner.

So the hole is **structurally documented** rather than fixed:
- An open issue (this file) names it, prioritizes it, and identifies the dependency.
- Two strict xfail tests pin the substrate's current behaviour; their flip to xpassed will signal that migration 002 has landed and the issue can be closed.

## Threat model — why HIGH not BLOCKER

Today's only writer is the `__cli__` role (T-12 application-layer enforcement). The CLI's deliberation-seal handler refuses operating on a sealed deliberation (`E_ALREADY_SEALED`); there is no code path through `selvedge` that can re-seal or mutate a sealed deliberation. The hole is exploitable only by:

1. A direct sqlite3 connection bypassing the CLI (debugging, ad-hoc, malicious).
2. A future submit kind written without these guards.
3. A future writer role added without trigger-level enforcement.

For (1), the workspace's defence is operator discipline + the audit pattern of running `bin/selvedge query` over `bin/selvedge`'s typed handlers. For (2) and (3), the next session that adds a writer or capability should treat OI-080-001 as a precondition. Migration 002 is the right structural remedy and must precede those additions.

## Closing condition

3. The two xfail tests in `test_deliberation_kinds.py` flip to xpassed and have been unmarked.
4. The closing decision records the migration's sha256 against the new `schema_migrations` row and confirms that the round-trip + concurrency test rerun against migration 001+002 still passes.
